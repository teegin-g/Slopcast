# Slopcast Refactor Plan

Created: `2026-04-04T19:33:36Z`

## Goal

Reduce architectural drift and maintenance risk in the repo without doing a broad, destabilizing rewrite.

This plan prioritizes:

- eliminating duplicate sources of truth
- normalizing API and persistence boundaries
- shrinking overly broad UI responsibilities
- improving backend lifecycle and validation hygiene

## What To Refactor First

### 1. Consolidate workspace state into one canonical path

Highest priority: `src/pages/SlopcastPage.tsx` and `src/hooks/useSlopcastWorkspace.ts` currently represent overlapping workspace orchestration paths.

Why this goes first:

- repo docs disagree about which path is canonical
- the two files compute similar economics and persistence behavior differently
- future fixes can land in one path and silently miss the other

Target outcome:

- one canonical workspace state owner
- page becomes composition/orchestration only
- shared helpers for storage, derived economics, workflow state, and export handlers exist once

## Execution Principles

- follow phased execution; keep each phase to `<= 5` touched files where possible
- do dead-code cleanup before structural refactors on large files
- prefer extraction and consolidation over rewrites
- keep compatibility shims temporarily when reducing drift
- verify behavior after each phase before moving on

## Phased Plan

### Phase 1. Identify and freeze the canonical workspace path

Files to focus on:

- `src/pages/SlopcastPage.tsx`
- `src/hooks/useSlopcastWorkspace.ts`
- `src/pages/FEATURE.md`
- `src/hooks/FEATURE.md`
- `src/components/slopcast/FEATURE.md`

Work:

- decide whether `SlopcastPage` should own state directly or delegate to a restored workspace hook
- mark the non-canonical path as legacy or remove it after extracting any missing behavior
- reconcile the folder docs so future agents and humans stop branching from conflicting assumptions

Success criteria:

- one clearly documented source of truth
- no duplicated `processedGroups` / persistence orchestration logic
- no conflicting guidance in local `FEATURE.md` files

### Phase 2. Extract storage and UI preference persistence

Likely files:

- `src/pages/SlopcastPage.tsx`
- `src/hooks/useSlopcastWorkspace.ts` or the new canonical workspace module
- `src/components/slopcast/hooks/useProjectPersistence.ts`
- `src/services/projectRepository.ts`
- one new module such as `src/services/storage/workspacePreferences.ts`

Work:

- centralize `localStorage` keys and migration rules
- stop scattering storage writes across page and hook layers
- create a thin facade for workspace UI preferences like tabs, FX mode, and panel state

Success criteria:

- storage keys are defined once
- storage migration logic is not duplicated
- page/component files no longer touch `localStorage` directly except through a facade

### Phase 3. Normalize frontend-to-backend API routing

Files to focus on:

- `server.js`
- `vite.config.ts`
- `src/services/economicsEngine.ts`
- `src/services/spatialService.ts`
- one targeted test file for API contract coverage

Work:

- make dev and production routing honor the same `/api/*` contract
- remove ambiguous proxy behavior around `/api`, `/api/engine`, economics endpoints, and spatial endpoints
- introduce a small integration check so route mismatches fail loudly

Success criteria:

- the same frontend request paths work in both Vite dev and production serving
- the API base path is defined once
- engine selection code does not embed environment-specific assumptions

### Phase 4. Reduce theme branching and push it to UI primitives

Best starting files:

- `src/components/ScenarioDashboard.tsx`
- `src/components/slopcast/SectionCard.tsx`
- `src/components/slopcast/PageHeader.tsx`
- `src/components/slopcast/DesignEconomicsView.tsx`
- one new theme-aware primitive module if needed

Work:

- remove repeated `theme.id === 'mario'` and `isClassic` branching from feature components where possible
- move classic-vs-default rendering differences into a small number of theme-aware primitives
- prefer theme tokens and `ThemeFeatures` over feature-level hard forks

Success criteria:

- fewer feature components contain explicit theme ID checks
- theme behavior is expressed through shared primitives or tokens
- adding a theme requires changing fewer feature files

### Phase 5. Split large multi-responsibility UI surfaces

Primary candidate:

- `src/components/ScenarioDashboard.tsx`

Secondary candidates after Phase 1:

- `src/pages/SlopcastPage.tsx`
- `src/components/slopcast/MapCommandCenter.tsx`
- `src/pages/IntegrationsPage.tsx`

Work:

- separate data preparation from rendering
- separate chart sections, accordions, and theme wrappers
- keep each extracted module focused on one concern

Success criteria:

- chart/data logic is testable without rendering the entire page
- presentational sections become smaller and easier to reason about
- large files stop mixing domain logic, state, layout, and theme branching

### Phase 6. Split `src/types.ts` into domain modules

Files to focus on:

- `src/types.ts`
- new files under `src/types/`
- a small number of import-heavy dependents per phase

Work:

- move domain types into modules such as `wells`, `economics`, `deals`, `integrations`, and `spatial`
- keep `src/types.ts` as a temporary re-export barrel during migration
- migrate imports gradually to avoid a giant flag day

Success criteria:

- unrelated type edits stop colliding in one file
- ownership by domain is clearer
- public imports continue working during transition

### Phase 7. Refactor backend service lifecycle and validation coverage

Files to focus on:

- `backend/spatial_service.py`
- `backend/main.py` or the FastAPI app entrypoint
- `backend/tests/test_spatial_routes.py`
- `.agents/validation/gate.sh`
- `backend/requirements.txt`

Work:

- replace module-level Databricks connection globals with an app-scoped provider or dependency
- make tests independent of mutable module state
- add backend validation coverage to the repo gate
- pin backend dependencies or adopt a lockfile strategy

Success criteria:

- backend services have explicit lifecycle management
- backend tests can run without hidden global resets
- the main validation gate exercises Python checks in addition to frontend checks

## Defer For Now

Do not start by refactoring `src/utils/economics.ts`.

Reason:

- it is large, but it appears to be concentrated pure logic rather than the main source of architectural drift
- the repo gets more risk reduction from consolidating workspace state and normalizing boundaries first

## Recommended Order

1. Phase 1: canonical workspace path
2. Phase 2: storage and persistence facade
3. Phase 3: API routing normalization
4. Phase 4: theme branching reduction
5. Phase 5: large UI surface decomposition
6. Phase 6: type module split
7. Phase 7: backend lifecycle and validation

## Verification After Each Phase

- `npm run typecheck`
- `npm test`
- `npm run build`
- `npx eslint . --quiet` if the repo lint config supports it cleanly
- for UI-touching phases: `npm run ui:audit`
- for route or workflow changes: `npm run ui:verify`
- for backend-touching phases: add and run `pytest backend/`

## Definition Of Done

This refactor effort is successful when:

- the repo has one obvious workspace state owner
- storage and API boundaries are centralized
- theme behavior is mostly token- or primitive-driven
- oversized UI files have narrower responsibilities
- backend services and validation are more explicit and reproducible
