## Anti-Patterns Verdict

### Does this look AI-generated? **Partial fail.**

The app avoids the worst AI slop tells (no gradient text, no hero metric grids with rounded cards, no "powered by AI" badges). The theme system is genuinely distinctive. However:

| AI Tell | Present? | Evidence |
|---------|----------|----------|
| Uniform glassmorphism | **YES** | Every non-Classic panel renders as `backdrop-blur-sm bg-*/60-80` regardless of theme. Slate (solid) and Synthwave (outline) still look glassy. |
| Dark mode + glowing accents | **Partial** | Intentional for Synthwave/Hyperborea, but the glow bleeds into Slate/Stormwatch where it doesn't belong. |
| Identical card grids | **Partial** | Economics KPIs are evenly-weighted 2x2 grids with no visual priority between NPV and secondary metrics. |
| Generic spacing | **NO** | Stormwatch dense spacing and per-theme radius are genuinely differentiated. |
| Decorative blur without purpose | **YES** | `PageHeader` blurs on all themes including Stormwatch (which wants dense/solid). |

**Verdict:** The *concept* is anti-slop. The *execution* leaks slop through hardcoded glass. Fix the consistency layer and this passes.

## Overall Impression

Slopcast's design ambition is real — seven themes with structural differentiation (typography, spacing, panel style, border radius) is far beyond what most apps attempt. The map overlays are the best-executed layer, correctly consuming `overlayPanelClass`. But **the economics views, headers, and mobile panels bypass the theme system entirely**, creating a paradox: the theme selector changes colors but panels all look the same.

The single biggest opportunity is **making `panelStyle` actually work** across every surface. This would instantly differentiate Slate (opaque, solid, utilitarian) from Synthwave (transparent, outlined, neon) from Tropical (frosted glass, organic) — delivering on the promise the theme system already architecturally supports.

## What's Working

### 1. Map overlays are theme-native
All 7 map overlay components (`OverlayGroupsPanel`, `OverlayFiltersBar`, `OverlayToolbar`, `OverlaySelectionBar`, `OverlayLegend`, `MapWellTooltip`, `WellPopupCard`) correctly consume `overlayPanelClass(theme.features.panelStyle)`. Switching from Slate to Synthwave visually transforms the map panels from solid to outlined. This is the gold standard the rest of the app should follow.

### 2. Classic theme hard-fork is clean
The `isClassic` conditional pattern is consistent: Classic gets `sc-panel` / `sc-header` / `sc-*` classes, modern themes get token-driven styles. This bifurcation is architecturally sound and avoids the trap of trying to make Classic "just another theme."

### 3. Per-theme typography creates real identity
Heading fonts (Orbitron for Synthwave, serif for Nocturne, monospace for Stormwatch) create instant recognizability beyond color changes. This is genuine structural differentiation.

## Priority Issues

### [P0] SectionCard and economics panels ignore `panelStyle`

**What:** `SectionCard` — the wrapper for every economics panel — defaults `panelStyle` to `'glass'` and uses its own `sectionStyleMap` that diverges from `overlayPanelClass`. Most callers don't pass `panelStyle` at all, so every panel renders as glass regardless of theme.

**Files:**
- `src/components/slopcast/SectionCard.tsx:16-20` — `sectionStyleMap` omits `backdrop-blur` for glass, uses different opacity values than `overlayPanelClass`
- `src/components/slopcast/SectionCard.tsx:32` — default `panelStyle = 'glass'`
- `src/components/slopcast/DesignEconomicsView.tsx` — no `panelStyle` prop passed to any `SectionCard`
- `src/components/slopcast/ScenarioDashboard.tsx` — same

**Why it matters:** This is the #1 reason themes feel like "color swaps." Stormwatch should have solid, opaque panels. Synthwave should have thin outlined panels. Both get glass.

**Fix:**
1. Delete `sectionStyleMap` from SectionCard
2. Import and use `overlayPanelClass` from `themes.ts` (single source of truth)
3. Change default from `'glass'` to consuming `theme.features.panelStyle` via `useTheme()`
4. All callers automatically get the correct style without changes

**Suggested command:** `/normalize`

### [P0] 21 hardcoded `backdrop-blur-*` instances across 14 files

**What:** Components apply `backdrop-blur-sm` or `backdrop-blur-md` directly in className strings, bypassing the theme system's `panelStyle`. Themes with `panelStyle: 'solid'` (Slate, Stormwatch) or `'outline'` (Synthwave, Nocturne) still render with frosted glass.

**Files (21 occurrences):**

| File | Count | Context |
|------|-------|---------|
| `DesignWellsView.tsx` | 4 | Mobile tabs, panel wrappers |
| `AcreageSearchBar.tsx` | 3 | Search dropdown |
| `MapVisualizer.tsx` | 3 | Inline map overlays |
| `PageHeader.tsx` | 1 | Main app header |
| `DesignEconomicsView.tsx` | 1 | Mobile panel switcher |
| `MapCommandCenter.tsx` | 1 | Command center panel |
| `HubPage.tsx` | 1 | Hub header |
| `IntegrationsPage.tsx` | 1 | Integration panels |
| `Controls.tsx` | 1 | Legacy controls |
| `Toast.tsx` | 1 | Toast notifications |
| `AiAssistant.tsx` | 1 | AI panel |
| `ProjectSharePanel.tsx` | 1 | Share dialog |
| `KeyboardShortcutsHelp.tsx` | 1 | Shortcuts modal |
| `ProfileSelector.tsx` | 1 | Profile dropdown |

**Why it matters:** Every `backdrop-blur` is a visual lie — it says "this is glass" even when the active theme says "this is solid." It's the CSS equivalent of ignoring the design system.

**Fix:** Replace all 21 occurrences with `overlayPanelClass(theme.features.panelStyle)` or, for components already using `SectionCard`, fix SectionCard (P0 above) and let it cascade.

**Suggested command:** `/normalize`

### [P1] PageHeader hardcodes glass styling

**What:** `PageHeader.tsx:50` applies `backdrop-blur-md border-b shadow-sm bg-theme-surface1/80` to the non-Classic header. This is glass styling regardless of theme.

**File:** `src/components/slopcast/PageHeader.tsx:49-51`
```tsx
isClassic ? 'sc-header' : 'backdrop-blur-md border-b shadow-sm bg-theme-surface1/80 border-theme-border'
```

Additionally, line 85 hardcodes `border-white/5` which doesn't adapt to theme context.

**Why it matters:** The header is the most persistent UI element — users see it on every page. If it doesn't change with the theme, the theme switch feels shallow.

**Fix:** Branch on `panelStyle`:
- `glass` → keep `backdrop-blur-md bg-theme-surface1/80`
- `solid` → `bg-theme-surface1 border-theme-border`
- `outline` → `bg-theme-surface1/20 border-theme-border/60`

Or create a `headerPanelClass()` variant of `overlayPanelClass` that uses `backdrop-blur-md` for glass headers.

**Suggested command:** `/normalize`

### [P1] Two separate header implementations

**What:** `HubPage.tsx:101` renders its own inline header with plain `<button>` elements and `backdrop-blur-sm`, separate from `PageHeader.tsx`. The HubPage header has different spacing, button styles, and doesn't use `AnimatedButton`.

**Files:**
- `src/pages/HubPage.tsx:101+` — inline header
- `src/components/slopcast/PageHeader.tsx` — primary header component

**Why it matters:** Users navigating between the Hub and Slopcast workspace see two visually different headers. This breaks spatial consistency and suggests two different apps.

**Fix:** Either extract a shared `AppHeader` that both pages compose, or have HubPage import and configure `PageHeader`.

**Suggested command:** `/extract`

### [P2] Hardcoded semantic colors bypass theme tokens

**What:** 20+ instances of raw Tailwind colors (`text-red-400`, `bg-emerald-500/90`, `text-green-400`) used where theme tokens should be.

**Key offenders:**

| File | Line | Hardcoded | Should be |
|------|------|-----------|-----------|
| `CashFlowTable.tsx` | 35 | `text-red-400` | `text-theme-danger` |
| `EconomicsGroupBar.tsx` | 417 | `bg-emerald-500/90` | `text-theme-success` or `bg-theme-positive` |
| `GroupList.tsx` | 152 | `text-red-400` | `text-theme-danger` |
| `OverlayToolbar.tsx` | 137 | `bg-green-500/20 text-green-300` | Theme success tokens |
| `IntegrationsPage.tsx` | 60+ | `text-green-400`, `text-red-400` | Theme semantic tokens |
| `MapCommandCenter.tsx` | 805, 811 | `text-red-400` (both branches!) | `text-theme-danger` |

**Why it matters:** Hardcoded `red-400` looks wrong against Nocturne's warm amber palette. These semantic colors should shift with the theme's personality — a "danger" state in Hyperborea should feel cold, not warm.

**Fix:** Define `--danger`, `--success`, `--warning` CSS custom properties per theme if they don't already exist (check `theme.css`), then replace raw colors.

**Suggested command:** `/colorize`

---

### [P2] Economics KPIs have flat visual hierarchy

**What:** The economics results display NPV10, IRR, EUR, payout months, and secondary metrics in evenly-weighted cards. All use the same size, weight, and prominence. NPV10 — the single most important number in O&G deal evaluation — has the same visual weight as a secondary metric like gas EUR.

**Why it matters:** Brand principle #4: "Data has gravity." NPV, IRR, and EUR are the stars — but the current layout gives them no gravitational pull. A deal analyst's eye should snap to NPV10 without scanning.

**Fix:**
1. Make NPV10 the hero metric — larger font, accent color, top-left position
2. IRR and EUR as primary supporting metrics — slightly smaller but still prominent
3. Payout, gas EUR, and secondary metrics in a supporting row with reduced visual weight
4. Consider a `/typeset` pass to establish a clear size ramp: hero (28px+) → primary (20px) → secondary (14px)

**Suggested command:** `/arrange` then `/typeset`

---

### [P2] Mobile tab bars duplicated across views

**What:** `DesignWellsView.tsx` (lines 220-261) and `DesignEconomicsView.tsx` (lines 161-190) both render nearly identical mobile tab bar UIs inline — same markup structure, same button styling patterns, different content. Neither extracts to a shared component.

**Why it matters:** Duplicated UI means duplicated bugs. When one gets a theme fix, the other stays broken. Both currently hardcode glass styling independently.

**Fix:** Extract a `MobileTabBar` component that both views compose.

**Suggested command:** `/extract`

---

### [P3] `glass.css` sidebar tokens bypass semantic system

**What:** `src/styles/glass.css` defines per-theme sidebar glass colors using raw `rgba()` values outside the CSS custom property system. These tokens don't respond to `panelStyle` and create a third styling path (alongside `overlayPanelClass` and `sectionStyleMap`).

**Why it matters:** Three separate glass/panel systems make theme maintenance exponential. Adding a new theme requires updating `themes.ts`, `theme.css`, AND `glass.css`.

**Fix:** Migrate `glass.css` tokens into the `[data-theme]` blocks in `theme.css` as CSS custom properties, then consume them like any other token.

**Suggested command:** `/normalize`

## Persona Red Flags

### Morgan — A&D Analyst (Power User)
Primary action: Run economics on a well group and compare NPV across scenarios.

- **KPI hierarchy is flat** — Morgan scans for NPV10 first, always. Having to visually parse identical-weight cards wastes 2-3 seconds per evaluation, and that adds up across 50 deals.
- **No keyboard shortcuts for scenario switching** — Morgan runs the same comparison across 3 pricing decks. No way to cycle scenarios without mouse.
- **"Compute: Fresh" badge uses raw green** — On Nocturne theme, `bg-emerald-500/90` clashes with the warm amber palette. Morgan doesn't consciously notice, but it registers as "this feels off."

### Jamie — Junior Engineer (First-Timer)
Primary action: Open Slopcast, select wells on map, see economics results.

- **All panels look the same** — Glass-on-glass-on-glass provides no spatial hierarchy. Jamie can't distinguish "this is the main content" from "this is a sidebar tool." The map overlays and economics panels have identical visual treatment.
- **No empty state guidance in economics** — Before running economics, the KPI cards show `$0` or blank. No copy like "Select wells and run economics to see results."
- **Mobile tab switching is modal** — On mobile, Jamie switches between SETUP and RESULTS tabs. The tab bar provides no visual indicator of whether data exists in the other tab, creating anxiety about losing work.

### Reyes — VP/Decision Maker (Glancer)
Primary action: Open a shared project link, scan economics, make a go/no-go call in 60 seconds.

- **NPV doesn't pop** — Reyes has 60 seconds. NPV10 should be screaming from the page. Instead it's one of six equally-styled numbers.
- **Theme inconsistency undermines trust** — If Reyes opens Slopcast on Stormwatch (solid, utilitarian) and sees glass blur effects, it reads as "unfinished" — the visual language is contradicting itself.
- **No print/export affordance visible** — Reyes wants to screenshot or export. No visible action for this in the economics view header.

## Minor Observations

1. **`MapCommandCenter.tsx:805`** — Both `isClassic` branches use identical `text-red-400`. The ternary is dead code.

2. **`AppShell.tsx`** — Built but never wired into production routes. Only referenced in test files. This is dead code that should be removed or integrated.

3. **`SectionCard` glass entry omits `backdrop-blur`** — `sectionStyleMap.glass` uses `bg-theme-surface1/72` without blur, while `overlayPanelClass` glass uses `backdrop-blur-sm bg-[var(--surface-1)]/80`. These are supposed to be the same style — they're visually different.

4. **Logo fallback uses DOM manipulation** — `PageHeader.tsx:71-81` uses `document.createElement` inside an `onError` handler instead of React state. This bypasses React's rendering model and could cause hydration issues.

5. **`border-white/5` in header** — `PageHeader.tsx:85` hardcodes a white-based border that won't adapt to themes with non-dark backgrounds (Slate light mode).
